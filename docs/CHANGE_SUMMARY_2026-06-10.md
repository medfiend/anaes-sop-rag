# AnaesSOP — Security & Quality Pass (2026-06-10)

Handoff summary of a coordinated security/quality refactor. Written so that any
agent or developer (e.g. Antigravity) can understand the current state of the
codebase without re-deriving it.

## Project context

AnaesSOP is a Next.js 16 (App Router, Turbopack) PWA for an anaesthetics
department: searchable clinical guideline database, emergency protocol
quick-access (zero-auth), admin PDF ingestion (Next.js → Cloudflare Worker →
Llama-3-8b compilation → Qwen embeddings → R2/D1), auto-generated dose
calculators, trust phonebook, offline guideline pinning, and an MTIR study
analytics endpoint. Auth is Clerk, restricted to NHS email domains, with a
passcode-based "demo mode" used for the current pilot.

---

## 1. Security hardening

### 1.1 Cloudflare Worker `/ingest` is no longer open to any Clerk user
`cloudflare-ai-worker/src/index.ts`

Previously any cryptographically valid Clerk JWT (i.e. any signup, any email
domain) could POST `/ingest` and publish/supersede live clinical guidelines.
Authorization is now one of, in order:

1. **Demo passcode** — `Bearer <DEMO_PASSCODE>` when `DEMO_MODE === 'true'`
   (unchanged; keeps the pilot demo working).
2. **Backend shared secret** — request header `X-Worker-Secret` equal to the
   worker secret `WORKER_SHARED_SECRET`. This is the intended production path:
   the Next.js `/api/upload` route enforces `requireAdmin` and then forwards
   this header (see `app/api/upload/route.ts`).
3. **Direct admin JWT** — a verified Clerk JWT whose `email` claim is on the
   admin allowlist (`ADMIN_EMAILS` worker var, comma-separated; defaults to
   `audit.lead@nhs.net,s.parashar1@nhs.net`). Note: default Clerk session
   tokens do NOT carry an email claim, so this path requires a customised JWT
   template — it is a deliberate fail-closed default.

**Deployment prerequisite:** generate one random secret and set it in BOTH
places, then redeploy the worker:

```
cd cloudflare-ai-worker
npx wrangler secret put WORKER_SHARED_SECRET   # paste the value
npx wrangler deploy
```

Vercel → Project Settings → Environment Variables → add
`WORKER_SHARED_SECRET` with the same value (and to local `.env.local`).
Until it is set, demo-mode uploads still work via path 1; non-demo uploads
will 401 at the worker.

### 1.2 NHS-domain policy enforced server-side
`lib/authGuard.ts`

- `requireAuth` now rejects authenticated users whose email is not
  `@nhs.net` / `*.nhs.uk` / on `ADMIN_EMAILS` (HTTP 403). Previously this rule
  existed only in client-side UI.
- Admin list reads from `process.env.ADMIN_EMAILS` (comma-separated) with the
  previous two emails as fallback.
- When the cookie session lacks an email claim, the guard now resolves the
  email via the Clerk backend API (`users.getUser`) instead of accepting
  the placeholder string `'authenticated-user'`.
- Demo cookie is parsed with exact name=value matching (was a substring
  `String.includes`, which matched passcode prefixes).
- **Latent bug fixed:** `clerk.verifyToken(...)` does not exist on
  `ClerkClient` in `@clerk/backend` v3 — bearer verification had been
  throwing and silently falling through to cookie auth. Now uses the
  standalone `verifyToken(token, { secretKey })` export.

### 1.3 Previously unauthenticated routes now guarded
- `app/api/guidelines/route.ts` — GET now requires `requireAuth`. (Emergency
  bypass is unaffected: it uses the static guidelines bundled client-side.)
- `app/api/analytics/mtir/route.ts` — GET and POST now require `requireAuth`.
  POST previously allowed anonymous writes into the MTIR study results table.
- `app/api/auth-check/route.ts` — **deleted**. It was an unauthenticated
  debug endpoint that leaked Clerk config including the secret-key prefix.

## 2. XSS and unsafe code execution removed

### 2.1 HTML-escaped markdown rendering
`lib/markdownFormat.ts` (new)

`formatMessageText` was inlined in `app/page.tsx` and regex-converted
markdown to HTML without escaping — LLM output, PDF-derived guideline text
and user queries flowed into `dangerouslySetInnerHTML` raw (stored-XSS path).
The shared helper now escapes `& < > " '` BEFORE applying the markdown
transforms, so only markup generated by the transforms themselves reaches the
DOM. Same CSS classes, so visual output is unchanged.

### 2.2 Safe formula evaluator replaces `new Function()`
`lib/safeFormula.ts` (new), used by `components/DoseCalculator.tsx`

Calculator schemas are LLM-generated (untrusted). Formulas were string-
substituted and executed with `new Function()` — an arbitrary-JS path.
`safeFormula.ts` implements a tokenizer + recursive-descent parser supporting
exactly: numbers, string literals, scope identifiers, `+ - * / %`,
comparisons (`> < >= <= === !== == !=`, equality always strict), `&& || !`,
ternary, parentheses, and an allowlist of `Math` functions
(`abs ceil floor round trunc sign sqrt pow min max`). Everything else throws
`FormulaError` (rendered as `Error` in the UI).

`evaluateCalculations(calculations, inputs)` evaluates the list **in order,
feeding each result back into scope** — this also fixes a real pre-existing
bug: dependent formulas (dexmedetomidine `ibw → bmi → dosing_weight → doses`)
previously evaluated to `'Error'` because earlier results were never added to
the evaluation scope.

Tests: `tests/safeFormula.test.ts` (11 node:test cases — run `npm test`):
arithmetic/precedence, ternaries, gender-based IBW dosing, Math allowlist,
injection rejection (`fetch`, `constructor.constructor`, template literals,
assignment, member access), dependent-calculation chains, `'AVOID'` outputs.

## 3. Honest search labelling (partial — approval gate deferred)

`app/hooks/useSearch.ts`, `app/page.tsx`

- Fabricated "Confidence Match: NN%" figures removed. Results carry
  `matchStrength: 'strong' | 'partial' | 'weak'` derived from Orama BM25
  score bands (`SCORE_*_THRESHOLD` constants), displayed via
  `MATCH_STRENGTH_LABELS` ("Strong keyword match", etc.).
- Hardcoded "100%" claims replaced with factual text: "selected directly from
  search results" / "pinned guideline".
- NOT yet done (explicitly deferred by the owner): human approval gate before
  LLM-generated calculators go live (`page.tsx` still renders
  `<DoseCalculator isApproved={true}>`; the component already supports a
  sandbox/approve flow used by the admin page).

## 4. Mock-vector search path removed

`app/hooks/useSearch.ts` rewritten; `app/api/embed/route.ts` **deleted**.

Previously: client docs were indexed with deterministic sin-hash **fake**
vectors while queries used real Qwen embeddings from `/api/embed` (which
itself silently returned a fake vector on failure) — cosine similarity
between those is meaningless. Search is now honest keyword BM25 with field
boosts (title ×3, synonyms ×2). The ingestion worker still writes real
per-record Qwen embeddings into `index/<id>_master.json` in R2, so a genuine
vector/hybrid path can be reintroduced later. `executeSearch(query)` lost its
second `skipEmbedding` parameter and is now memoized with `useCallback`.

## 5. Quality pass

- **Expo/React Native fully removed.** Deleted: `app_expo/`,
  `components/LocationSelector.tsx`, `app.json`, `babel.config.js` (Next now
  uses SWC), `static_mock/`, `vercel.bak`, compiled pipeline artifacts
  (`pipeline/{compile,audit,run_compile}.js`). `package.json` dropped expo*,
  react-native*, babel deps and unused `minisearch`/`@types/jest`; added
  explicit `@clerk/backend`. Scripts are now `dev/build/start/test/compile`.
- **`components/DoseCalculator.tsx` rewritten in plain React + Tailwind**
  (same tape-scroller UX, CSS scroll-snap instead of RN ScrollView).
- **TypeScript build errors enabled**: `ignoreBuildErrors` removed from
  `next.config.js`; new standalone `tsconfig.json` (no `expo/tsconfig.base`),
  excludes `cloudflare-ai-worker` and `pipeline` (separate toolchains).
  `next build` passes clean.
- **Dedupe:** one `buildIndex()` in `useSearch` (was two identical ~60-line
  blocks), one `WELCOME_MESSAGE` constant in `page.tsx` (was three copies),
  removed unused mock imports from `page.tsx`.

## Environment variable reference (new/changed)

| Variable | Where | Purpose |
|---|---|---|
| `WORKER_SHARED_SECRET` | Vercel + worker secret | Server-to-server auth for `/ingest` (required for non-demo uploads) |
| `ADMIN_EMAILS` | Vercel + worker var (optional) | Comma-separated admin allowlist; falls back to the two hardcoded emails |
| `DEMO_MODE`, `DEMO_PASSCODE` | unchanged | Demo bypass; ensure NOT set in real production |

## Known remaining issues / backlog (not addressed in this pass)

1. Calculator human-approval gate before publication (deferred by owner).
2. Citations use fake page-1/highlight rectangles; real coordinates should be
   extracted at ingestion.
3. Trust SOP PDFs in `public/` are served unauthenticated (bypassing
   `/api/pdf` auth); `data/phonebook.json` (real staff names/extensions) is
   bundled into the client and committed to git — IG review needed.
4. Demo mode grants full admin; `NEXT_PUBLIC_DEMO_PASSCODE` has a hardcoded
   fallback (`NHS2026`) visible in the client bundle.
5. Worker sends the full PDF text to Llama-3-8b without chunking; long
   documents overflow context and drop to a cruder fallback.
6. Upload route: telemetry hardcoded to zeros; D1 rows stuck at status
   `'uploading'` if the worker fails are never cleaned up; response claims
   `text/event-stream` but emits NDJSON.
7. `app/page.tsx` (~1,200 lines) and `app/admin/page.tsx` (~1,280 lines)
   should be split into components.
8. No CI; only `tests/safeFormula.test.ts` exists.
